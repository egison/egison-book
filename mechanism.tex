\chapter{Egisonパターンマッチの仕組み}\label{mechanism}

本章は，Egison内部のパターンマッチの仕組みを解説する．
まず，\ref{mechanism-overview}節で概略を説明する．
Egisonの使い方に興味ある読者はこの節を読めば，第\ref{matcher}章に進むことができる．
\ref{mechanism-traversal}節以降では，Egisonパターンマッチを自身で実装するためには必要な事項を解説する．

\section{パターンマッチ・アルゴリズムの概略}\label{mechanism-overview}

以下の\lstinline{matchAll}式を実行したときにどのような処理がなされるのかみていくことにより，Egison内部のパターンマッチ・アルゴリズムの概略をつかむ．
\begin{lstlisting}[language=egison,numbers=none]
matchAll [2,8,2] as multiset eq with
| $m :: #m :: _ -> m
-- [2,2]
\end{lstlisting}

上記の\lstinline{matchAll}式を受け取るとEgison処理系は，以下のようなマッチング・ステート（\emph{matching state}）と呼ばれるオブジェクトを処理系内部で生成する．
マッチング・ステートは，マッチング・アトム（emph{matching atom}）のスタックと，パターンマッチの途中結果からなる．
マッチング・アトムは，パターン，マッチャー，ターゲットからなる3つ組である．
初期マッチング・ステートは，\lstinline{matchAll}式のパターン，マッチャー，ターゲットから構成されるマッチング・アトム1つからなるスタックから構成される．
Egison内部のパターンマッチ・アルゴリズムは，マッチング・ステートの還元プロセスとして定義されている．
還元の結果，マッチング・アトムのスタックが空になるとパターンマッチに成功する．
\begin{lstlisting}[language=egison,numbers=none]
MState [($m :: #m :: _, multiset eq, [2,8,2])]
       []
\end{lstlisting}
このマッチング・ステートは，次のステップで以下の3つのマッチング・ステートに還元される．
この還元は，\lstinline{multiset}マッチャーのコンス・パターンの定義（第\ref{matcher}章\ref{matcher-multiset}節にて解説する）にもとづいて，実行される．
\begin{lstlisting}[language=egison,numbers=none]
MState [($m, eq, 2)
       ,(#m :: _, multiset eq, [8,2])]
       []

MState [($m, eq, 8)
       ,(#m :: _, multiset eq, [2,2])]
       []

MState [($m, eq, 2)
       ,(#m :: _, multiset eq, [2,8])]
       []
\end{lstlisting}
このように1つのマッチング・ステートを1ステップ還元すると複数のマッチング・ステートが生成される．
生成されるマッチング・ステートの数が0個であることも，無限個ある場合もある．
これらのマッチング・ステートがどのような順番で還元されるのかは，\ref{mechanism-traversal}節で論じる．
ここでは，1つ目のマッチング・ステートの還元をみる．
1つ目のマッチング・ステートは，次のステップで以下のように還元される．
スタックの先頭のマッチング・アトムのマッチャーが\lstinline{eq}から\lstinline{something}に変わる．
この還元は，\lstinline{eq}マッチャーに定義されている．（第\ref{matcher}章\ref{matcher-eq}節でその定義をみる．）
\begin{lstlisting}[language=egison,numbers=none]
MState [($m, something, 2)
       ,(#m :: _, multiset eq, [8,2])]
       []
\end{lstlisting}
\lstinline{something}はEgison唯一の組み込みマッチャーであり，パターンマッチの途中結果に新しい束縛を追加する．
ここでは，変数\lstinline{m}に\lstinline{2}を束縛している．
\begin{lstlisting}[language=egison,numbers=none]
MState [(#m :: _, multiset eq, [8,2])]
       [(m, 2)]
\end{lstlisting}
再び，\lstinline{multiset}のコンス・パターンの定義にもとづき，マッチング・ステートが還元される．
\begin{lstlisting}[language=egison,numbers=none]
MState [(#m, eq, 8)
       ,(_, multiset eq, [2])]
       [(m, 2)]

MState [(#m, eq, 2)
       ,(_, multiset eq, [8])]
       [(m, 2)]
\end{lstlisting}
上記，1つ目のマッチング・ステートは値パターン\lstinline{#m}のパターンマッチに失敗して消える．（還元された結果0個のマッチング・ステートを生成するともいうことができる．）
2つ目のマッチング・ステートは，先頭のマッチング・アトムがポップアウトし，以下のように還元される．
\begin{lstlisting}[language=egison,numbers=none]
MState [(_, multiset eq, [8])]
       [(m, 2)]
\end{lstlisting}
このマッチング・ステートも，先頭のマッチング・アトムがポップアウトし，以下のように還元される．
\begin{lstlisting}[language=egison,numbers=none]
MState []
       [(m, 2)]
\end{lstlisting}
最終的にマッチング・アトムのスタックが上記のように空になると，このパターンマッチの途中結果が最終結果としてボディの評価に使われる．

\section{\texttt{matchAll}・\texttt{matchAllDFS}による探索木のトラバース}\label{mechanism-traversal}

本節では，\lstinline{matchAll}式と\lstinline{matchAllDFS}式を実行したときに内部でどのような探索木がトラバースされるのかを解説する．
ある1つのマッチング・ステートを還元すると0個から無限個までを含む複数のマッチング・ステートが生成される．
そのため，Egisonの内部のパターンマッチ・アルゴリズムは，初期マッチング・ステートをルートとするツリーの探索と考えることができる．

まずは，以下のような\lstinline{matchAllDFS}式の探索木をみる．
\begin{lstlisting}[language=egison,numbers=none]
take 8 (matchAllDFS [1..] as set integer with
        | $m :: $n :: _ -> (m, n))
-- [(1, 1), (1, 2), (1, 3), (1, 4), (1, 5), (1, 6), (1, 7), (1, 8)]
\end{lstlisting}
この\lstinline{matchAllDFS}式の探索木は図\ref{fig:reduction-tree-1}のようになる．
図中の四角は，1つのマッチング・ステートを表現している．
図中の丸は，マッチング・アトムのスタックが空になったマッチング・ステート，パターンマッチの最終結果を表現している．
\begin{figure}[t]
  \centering
  \includegraphics[width=6.0cm,bb=0 0 529 582]{reduction-tree-1.png}
  \caption{\lstinline{matchAllDFS}式の探索木}
  \label{fig:reduction-tree-1}
\end{figure}
\lstinline{matchAllDFS}式はこの探索木を深さ優先探索する．
そのため，マッチング・ステートは，a1，b2，e3，f4，h5，j6，m7，o8，k7，p8，r9，...という順番で還元されていく．
マッチング・ステートh5は無限個のマッチング・ステート（j6，k7，...）に還元される．
そのため，深さ優先探索では，マッチング・ステートc3の還元に行き着くことがない．
このように\lstinline{matchAllDFS}では，可算無限のパターンマッチ結果をすべて列挙できない場合がある．

\lstinline{matchAll}式は，可算無限のパターンマッチ結果をすべて列挙するために，この探索木に変形を加えて幅優先探索をおこなう．
図\ref{fig:reduction-tree-2}は，下記の\lstinline{matchAll}式を実行したときの探索木である．
\begin{lstlisting}[language=egison,numbers=none]
take 8 (matchAll [1..] as set integer with
        | $m :: $n :: _ -> (m, n))
-- [(1, 1), (1, 2), (2, 1), (1, 3), (2, 2), (3, 1), (1, 4), (2, 3)]
\end{lstlisting}
\begin{figure}[t]
  \centering
  \includegraphics[width=10.5cm,bb=0 0 929 669]{reduction-tree-2.png}
  \caption{\lstinline{matchAll}式の探索木}
  \label{fig:reduction-tree-2}
\end{figure}
図\ref{fig:reduction-tree-1}の探索木のについては，1つのマッチング・ステートが1つのノードをあらわしていたが，図\ref{fig:reduction-tree-2}の探索木のについては，一連なりのマッチング・ステートのリストを1つのノードとする．
たとえば，a1は1つのマッチング・ステートからなるノード，b2，c3，d4，...は無限個のマッチング・ステートからなる1つのノードである．
このように探索木をとらえると，この探索木は二分木になっている．
探索木が二分木であるため，すべてのノードの子が有限であるため，幅優先探索をすれば，すべてのノードをたどることができる．

図\ref{fig:reduction-tree-1}の探索木を斜めにとらえることによって，図\ref{fig:reduction-tree-2}の二分木への変形はできる．
b2，c3，d4，...からなるノードに注目しよう．
このノードの子は，e3単独からなるノードと，c3，d4，...からなるノードである．
一般に，あるノードの子は，そのノードの先頭のマッチング・ステートを還元した結果と，そのノードの先頭の要素を取り除いたマッチング・ステートのリストとなる．
